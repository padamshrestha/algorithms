# Ideas

This question looks very complicated, and you need to consider a lot of situations. , Belongs to the kind of question that has no technical content, but tests the programming ability, and requires sufficient thinking. This kind of simulated question is what the question tells me to do. Similar to the prisoner room problem written before, the Joseph ring is also a simulation, but you need to prun and optimize it after the simulation.

There are actually many situations for this question. We need to consider the difficulty of each set of questions, not the difficulty of different sets of questions. The question requires us to meet: a<=b<=c ba<=10 cb<=10, that is, after the difficulty of the question is sorted from small to large, the adjacent difficulty cannot be greater than 10.

Therefore, our idea is to sort first, then traverse from small to large, if the difficulty of the adjacent is not greater than 10, then continue. If it is not met, we can only ask the byte teacher to ask a question to satisfy the condition.

Since only the difficulty of the same set of questions needs to be compared, my idea is to ** compare the diff of the second and the first, and the third and the second of the same set of questions**.

-If diff is less than 10, do nothing and continue. -If diff is greater than 10, we must add a question.

There are a few points to note here.

For the second question:

-For example, the difficulty is 1 30 40. I can add a 21 between 1, 30, so that 1, 21, 30 can form a set. -For example, the difficulty is 1 50 60. I can add 21, 41 between 1, 50 to form a set, and my own (50) cannot be grouped into this set of questions anyway.

It is not difficult to see that the critical point of the second question is diff = 20. If it is less than or equal to 20, you can group yourself into a set of questions, just add one, otherwise you need to add two, and you can't group yourself into the current set of questions.

For the third question:

-For example, 1 20 40. I can add a 30 between 20 and 40, so that 1, 20, 30 can form a set, and the self (40) cannot be combined into this set of questions. -For example, 1 20 60. The same is true. I can add a 30 between 20 and 60, and the self (60) cannot be grouped into this set of questions.

It is not difficult to see that the critical point of the third question is diff = 10. If it is less than or equal to 10, you can group yourself into a set of questions, otherwise you need to add one, and you cannot group yourself into the current set of questions.

This is all the case.

Some students are more curious about how I think. How do I guarantee that there is no repetition or omission.

In fact, this question is a decision tree, and you will understand if I draw a decision tree.

The red border in the picture indicates that it can form part of the set of questions, and I also explained it in words. #2 represents the second question, #3 represents the third question.

As you can see from the figure, I have considered all the cases. If you can draw this decision diagram like me, I think you will not miss it. Of course, my solution is not necessarily optimal, but it is indeed a very useful and universal thinking framework.

        if cur == 1:

        if cur == 2:
            cur = 1
        cnt += 1
    if diff> 20:

            cnt += 2
        if cur == 2:
            cnt += 1

# Pre-knowledge

-The nature of the exclusive OR operation -How to efficiently compare the size of two numbers (from high to low)

First, popularize the pre-knowledge. The first is the exclusive OR operation:

The property of XOR: the result of XOR of two numbers, a^b, is the number obtained by performing operations on each bit of the binary of a and b. The logic of the operation is 0 if the same digit is the same number, and 1 if it is different

The law of XOR:

  1. The exclusive OR of any number and itself is 0

  2. The exclusive OR of any number and 0

  3. The exclusive OR operation satisfies the commutative law, namely: a ^ b ^ c = a ^ c ^ b

At the same time, I suggest you to take a look at the classic topics of several bit calculations that I have summarized. [Bit Operation Series](https://leetcode-cn.com/problems/single-number/solution/zhi-chu-xian-yi-ci-de-shu-xi-lie-wei-yun-suan-by- 3/ "Bit Operation Series")

According to the XOR knowledge I mentioned earlier, it is not difficult to conclude that the XOR structure is 110.

I performed XOR three times above:

  1. The first time is the exclusive OR of the most significant 0 and 1, and the result is 1.
  2. The second time is the exclusive OR of the second highest 1 and 0, and the result is 1.
  3. The third time is the XOR of the lowest 1 and 1, and the result is 0.

So how is m 1? Do we need to perform XOR three times? In fact, when the first XOR is performed, it is known that it must be greater than m (m is 1). Because the structure of the first XOR results in its highest bit being 1, that is to say, its smallest value is only 100, which is 4, which must be greater than 1. This is pruning, which is the key to algorithm optimization.

Do you see my step-by-step thinking process? All algorithm optimization needs to go through a similar process.

Therefore, my algorithm is to XOR two pairs from the high bit, and the result of the XOR is compared with the binary bit corresponding to m.

-If it is larger or smaller than the binary digit corresponding to m, we can exit early. -If they are equal, we continue to move to the lower position and repeat the process.

**Analyze the algorithm, find the bottleneck part of the algorithm, and then select the appropriate data structure and algorithm to optimize it. ** This sentence is very important, please remember it.

Here, we are using pruning technology. Regarding pruning, the 91 Tianxue algorithm also has a detailed introduction.

Going back to the algorithm bottleneck mentioned earlier, multiple numbers have a common prefix, and the prefix part is the number of operations we wasted. When it comes to prefixes, everyone should be able to think of a prefix tree. If you are not familiar with prefix trees, take a look at my Prefix Tree Topic (opens new window), all the questions inside It's almost enough to write it once.

Therefore, one idea is to build a prefix tree, where the root of the tree is the highest bit. Since the title requires XOR, we know the bitwise operation of XOR binary, so it is better for this tree to store binary.

Take a backhand glance at the data range: m, n<=10^5. 10^5 = 2 ^ x, our goal is to find the ceil (rounded up) of x that satisfies the condition, so x should be 17.

Each node of the tree stores: ** among the n numbers, how many prefixes formed from the root node to the current node are the same **, that is, how many prefixes are the same. This can be pruned, and when you withdraw early, you can directly take it out and use it. For example, the result of the XOR is 1, and the current binary digit of m is 0, so there are 10 prefixes, and I don’t need to compare them. The counter is directly +10.

I used 17 directly and the complexity is too high, and currently only passed 70%-80% of the test cases, I hope everyone can help me find the problem, I guess it is the language of the pot.

# Code


class TreeNode:
    def __init__(self):
        self.cnt = 1
        self.children = [None] * 2
def solve(num, i, cur):
    if cur == None or i == -1: return 0
    bit = (num >> i) & 1
    mbit = (m >> i) & 1
    if bit == 0 and mbit == 0:
        return (cur.children[1].cnt if cur.children[1] else 0) + solve(num, i-1, cur.children[0])
    if bit == 1 and mbit == 0:
        return (cur.children[0].cnt if cur.children[0] else 0) + solve(num, i-1, cur.children[1])
    if bit == 0 and mbit == 1:
        return solve(num, i-1, cur.children[1])
    if bit == 1 and mbit == 1:
        return solve(num, i-1, cur.children[0])

def preprocess(nums, root):
    for num in nums:
        cur = root
        for i in range(16, -1, -1):
            bit = (num >> i) & 1
            if cur.children[bit]:
                cur.children[bit].cnt += 1
            else:
                cur.children[bit] = TreeNode()
            cur = cur.children[bit]

n, m = map(int, input().split())
nums = list(map(int, input().split()))
root = TreeNode()
preprocess(nums, root)
ans = 0
for num in nums:
    ans += solve(num, 16, root)
print(ans // 2)

This algorithm can pass 50% case.

The reason for the inefficiency of the above algorithm is to open up the N space and sort the entire N elements.

A simple optimization method is to replace the sorting with a heap, and use the characteristics of the heap to find the k-th largest number, so that the time complexity can be reduced to mlogNmlogNmlogN.

We continue to optimize. In fact, if you draw the lexicographic sorting structure, you can find that it is essentially a decimal tree, and it is a complete decimal tree.

Next, I will take you to continue the analysis.

As shown in the figure, red indicates the root node. The node represents a decimal number. The path of the tree stores the real number, such as 100, 109, etc. on the graph. Isn't this the prefix tree mentioned above?

The yellow part in the figure indicates the lexicographic order, pay attention to the direction of the arrow. Therefore, in essence, to find the m-th number in lexicographic order is to find the m-th node traversed in the preorder of this tree. **

Therefore, an optimization idea is to build such a tree and then traverse it. The complexity of construction is O(N)O(N)O(N), and the complexity of traversal is O(M)O(M)O(M). Therefore, the complexity of this algorithm can reach O(max(m,n))O(max(m, n))O(max(m,n)). Since n >= m, it is O(N)O(N)O(N).

In fact, such an optimized algorithm is still unable to test all AC test cases, and will exceed the memory limit. Therefore, our idea can only be to construct the tree without using the space of N. Think about it and also know that since N can be up to 10^18, a number is calculated as 4 bytes, then there are 400000000 bytes, which is about 381 M, which is unacceptable.

The problem mentioned above is the preorder traversal of a complete decatree, and the problem is transformed into finding the mth number of the preorder traversal of the complete decatree.

There is no essential difference between a decimal tree and a binary tree. In the topic of binary trees, I also mentioned that N-ary trees can be represented by binary trees.

For a node, the mth node:

-Either it is itself -Either in its child node -Either in its sibling node -Either in the child node of the sibling node

Which one of the above four parts is in depends on the number of its child nodes.

-count> m, m is in its child nodes, we need to drill down to the child nodes. -count <= m, m is not in itself and child nodes, we should skip all child nodes and go directly to sibling nodes.

This is essentially a recursive process.

It should be noted that we will not really walk on the tree, so how do we do the above-mentioned deep down to the child nodes, and skip all child nodes and go directly to the sibling nodes?

If you observe carefully, you will find: If the prefix of the current node is x, then its first child node (the smallest child node) is x * 10, the second is x * 10 + 1, and so on. therefore:

-Going down to the child node is x * 10. -Skip all child nodes, and go directly to the sibling node is x + 1.

Ok, the paving is almost complete.

Next, our focus is how to calculate the number of child nodes of a given node.

This process is no different from the number of nodes in a complete binary tree. The time complexity of this algorithm should be O(logN∗logN)O(logN*logN)O(logN∗logN). If you don’t know how to do it, you can refer to the original question of Likou: 222. Number of nodes in a complete binary tree (opens new window), this is a medium-difficulty problem.

So this question itself is divided into hard, which is not an exaggeration.

Here is a brief introduction, the idea of ​​calculating the number of child nodes of a given node, this problem has been presented in my 91 days learning algorithm.

A simple but non-optimal idea is to calculate the depth of the left and right subtrees separately.

-If the height of the left and right subtrees of the current node is the same, then the left subtree is a full binary tree, and the right subtree is a complete binary tree. -Otherwise (the height on the left is greater than the right), then the left subtree is a full binary tree, and the right subtree is a full binary tree.

If it is a full binary tree, the current number of nodes is 2 ^ depth, and for a full binary tree, we can continue recursion.

class Solution:
    def countNodes(self, root):
        if not root:
            return 0
        ld = self.getDepth(root.left)
        rd = self.getDepth(root.right)
        if ld == rd:
            return 2 ** ld + self.countNodes(root.right)
        else:
            return 2 ** rd + self.countNodes(root.left)

    def getDepth(self, root):
        if not root:
            return 0
        return 1 + self.getDepth(root.left)

And this question, we can be simpler and more efficient.

For example, we want to calculate the number of child nodes of node 1.

-The number of its child nodes is. . . -The number of its grandchildren is. . . -. . .

Add them all up.

The number of its child nodes is 20-10 = 10. That is, the first child node of its right sibling node minus its first child node.

Because it is a complete decree, not a full decree. So you need to consider the boundary conditions, for example, the n of the title is 15. Then the number of child nodes of 1 is not 20-10 = 10, but 15-10 + 1 = 16.

Others are similar processes, we only need to:

-Go deeper and do the same thing

or:

-Move to next neighbor and do the same thing

Keep repeating until m decreases to 0.

# Code


def count(c1, c2, n):
    steps = 0
    while c1 <= n:
        steps += min(n + 1, c2)-c1
        c1 *= 10
        c2 *= 10
    return steps
def findKthNumber(n: int, k: int) -> int:
    cur = 1
    k = k-1
    while k> 0:
        steps = count(cur, cur + 1, n)
        if steps <= k:
            cur += 1
            k -= steps
        else:
            cur *= 10
            k -= 1
    return cur
n, m = map(int, input().split())
print(findKthNumber(n, m))
